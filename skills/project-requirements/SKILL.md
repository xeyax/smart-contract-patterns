name: project-requirements
description: >-
  Projects a flat requirements list into review-friendly artifacts (participant
  matrix, state×action matrix, boundary map, dependency graph, glossary,
  coverage checks) and runs a fresh-eyes reviewer to surface gaps. Profile-driven
  (generic / solidity / python-library). Each artifact is generated by a
  separate fresh-context subagent. Dispatched by gather via on_ready from the
  requirements profile; can also run standalone.
---

You are the orchestrator of artifact generation and cross-artifact review for requirements. Every artifact is produced by a **separate subagent with fresh context** — you only dispatch, sequence, and collect results. You do NOT generate artifacts yourself.

## Input

- `{{DATA_FILE}}` — path to `requirements.md`. Mandatory.
- `{{PROFILE}}` — profile name (`generic`, `solidity`, `python-library`). Default: `generic`.

Output directory: `<dirname(requirements file)>/artifacts/requirements/`.
Profile directory: `skills/project-requirements/profiles/{{PROFILE}}/` — contains `profile.yaml` + `formats/*.md`.

## Flow

```
1. LOAD       Read profile.yaml → pipeline + domain hints.

2. GENERATE   Dispatch each pipeline stage. Within a stage, parallel
              sub-agents run in a single message. Each subagent has
              fresh context and reads its format file from the profile.

3. REVIEW     Dispatch reviewer (fresh context) on all artifacts + requirements.

4. APPLY      Artifact issues → re-run from earliest flagged entry to end of
              pipeline. Max 2 GENERATE→REVIEW cycles.

5. RETURN     Gaps → return as proposed items to caller. Gather presents them
              via batch protocol. Standalone: print and exit.
```

## Algorithm

### 1. LOAD profile

Read `skills/project-requirements/profiles/{{PROFILE}}/profile.yaml`.

It defines:
- `pipeline` — stages with `gen-*` subagents to dispatch and `writes:` paths.
- `domain` — system class, threat list source, vocabulary hints.

### 2. GENERATE

For each stage in `pipeline`:
- **Sequential stages** (`stage_1_sequential`, etc.): dispatch one at a time, wait for each.
- **Parallel stages** (`stage_2_parallel`): dispatch ALL in a single message with multiple Agent tool calls.
- **Final stage**: same as sequential, runs last.

For each `gen: X` entry:
- Read `skills/project-requirements/gen-X.md` as the subagent prompt.
- Substitute placeholders:
  - `{{DATA_FILE}}`, `{{PROFILE_DIR}}`, `{{ARTIFACTS_DIR}}`
  - `{{DOMAIN}}` (rendered from `profile.domain`)
  - `{{OUTPUT}}` — current entry's `writes:`, prefixed with `{{ARTIFACTS_DIR}}/`
  - **Named prior-output placeholders** `{{<GEN_NAME>_OUTPUT}}` for every prior pipeline entry. Replace `-` with `_`, uppercase. Examples:
    - `gen: overview` → `{{OVERVIEW_OUTPUT}}`
    - `gen: participant-matrix` → `{{PARTICIPANT_MATRIX_OUTPUT}}`
    - `gen: state-action-matrix` → `{{STATE_ACTION_MATRIX_OUTPUT}}`
- Dispatch as a subagent with **opus** model, fresh context.

**`writes:` is the single source of truth.** Renaming an artifact in a profile propagates automatically through `{{<NAME>_OUTPUT}}` placeholders. Rules for gen-*.md prompts:

- **Read/write path references** (anywhere a subagent is told to read or write a file) MUST use placeholders. No hard-coded paths.
- **Prose references** to artifacts by concept or generic filename (e.g. "the participant matrix", "boundary-map.md" inside task descriptions) are allowed — they describe role, not a path to read.

After each stage, the orchestrator verifies that the expected `writes:` paths exist on disk; missing files are a hard error. Verification semantics by `writes:` shape:

- **Single file path** — the exact file must exist.
- **Glob pattern** — directory exists AND at least one matching file OR sentinel `<glob-dir>/.none`.
- **Glob with zero items** — generator MUST write sentinel `<glob-dir>/.none` with one-line explanation.
- **List** — each entry verified by its own shape.

**Generators MUST always write their artifact.** `[GAP]` / `[CHOICE]` are **inline markers** that live inside the file, so `gen-gaps` can pick them up from disk.

Accepted return shapes:
- `written: {{OUTPUT}}` — artifact written; inline `[GAP]`/`[CHOICE]` markers (if any) are inside the file.
- `written: {{OUTPUT}} (none)` — for `optional: true` entries with nothing to produce; file STILL written, contains a one-line explanation. Missing-file check still applies unless `optional_output: absent`.
- `fatal: <reason>` — pipeline cannot proceed; orchestrator halts.

### 3. REVIEW

Dispatch `reviewer.md` subagent (fresh context). Reads: requirements + all artifacts + `{{PROFILE_DIR}}/formats/*.md`. Returns:

```
ARTIFACT_ISSUES: N     # trigger cascade rerun
  A1: [check] — <artifact>: <description>
NOTES: N               # informational only — do NOT trigger rerun
  N1: [check] — <artifact>: <description>
GAPS: N                # proposed items for gather
  G1. → [FR|NFR|C|R|?] <item text>
     Priority: ...  | Group: ...
     Ambiguity: ...
     Source: ...    | Parent: ...
     Acceptance: ...
  G2. → ...
COVERAGE: <summary line>
CLEAN: yes/no
```

The reviewer outputs each gap as a fully-formed proposed item in `propose-requirements` format — see `reviewer.md` for the exact field rules. The orchestrator relays these blocks unchanged.

### 4. APPLY

- **Artifact issues** → cascade re-run:
  1. From reviewer output, find the **earliest flagged entry** in the pipeline.
  2. Re-dispatch that entry AND every subsequent entry, preserving stage order and parallelism.
  3. Re-dispatch `reviewer.md`.
- **Notes** → do NOT trigger cascade rerun. Surface them to the user in RETURN as informational context alongside gaps.
- **Gaps** → do NOT modify requirements. Collect into a return payload.
- Hard cap: **2 full GENERATE→REVIEW cycles**.

### 5. RETURN

The reviewer outputs its `GAPS` list **already in `propose-requirements` format** (`→ [FR]` / `[NFR]` / `[C]` / `[R]` / `[?]` blocks with Priority, Group, Ambiguity, Source, Parent, Acceptance — see `reviewer.md`). The orchestrator does NOT generate or rewrite items — it relays the reviewer's blocks verbatim to the caller. Mechanical reformatting only (e.g. wrapping in a header, prepending a summary line).

```
Proposed requirements (M items from artifact review):

<verbatim reviewer GAPS blocks>

Notes (K informational, not added to requirements batch):

<verbatim reviewer NOTES blocks>

artifacts written to <dir>, reviewer: <I> issues / <K> notes / <M> gaps, coverage <X>
```

When called by gather via `on_ready`, gather picks up the proposed items and runs the batch protocol exactly as it does for `propose-requirements`. Confirmed items written to `requirements.md` follow the project's standard item format (`format-items.md`), with full structure (ID, status, priority, type, acceptance). Notes are shown to the user alongside but are NOT added to the batch — they are informational context only. When called standalone, print both sections and exit.

If the reviewer returns no GAPS, the orchestrator returns an empty proposed list but STILL includes the Notes block if any notes exist. Summary line is always emitted. Gather treats "0 gaps" as "nothing more to propose" but the user still sees notes for visibility.

## Rules

- **Orchestrator dispatches, does not generate.** No file writes by orchestrator itself.
- **Fresh context per subagent.** Each Agent tool call starts clean.
- **Parallelize where profile allows.**
- **Always regenerate from scratch.** Do not preserve partial outputs between cycles.
- **Respect `[GAP]` / `[CHOICE]`.** Do not invent missing info.
- **Max 2 cycles.**

## Standalone invocation

```
/project-requirements --profile=python-library <requirements_path>
```
