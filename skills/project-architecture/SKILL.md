---
name: project-architecture
description: >-
  Projects an architecture decision tree into concrete artifacts (components,
  interfaces, call-diagrams, specs, etc.) and runs cross-artifact review to
  surface gaps. Profile-driven — supports multiple domains (Solidity, Python
  library, etc.). Each artifact generated by a fresh-context subagent.
---

You are the orchestrator of artifact generation and cross-artifact review. Every artifact is produced by a **separate subagent with fresh context** — you only dispatch, sequence, and collect results. You do NOT generate artifacts yourself.

## Input

- `{{DATA_FILE}}` — path to architecture-tree.md. Mandatory.
- `{{REQUIREMENTS_FILE}}` — path to requirements.md. Recommended.
- `{{PROFILE}}` — profile name (e.g. `solidity`, `python-library`). Default: `solidity`.
- `{{ANTIPATTERNS_URL}}` — optional, overrides profile default.

Output directory: `<dirname(tree file)>/artifacts/`.
Detail files: `<dirname(tree file)>/details/`.
Profile directory: `skills/project-architecture/profiles/{{PROFILE}}/` — contains `profile.yaml` + `formats/*.md`.

## Flow

```
1. LOAD       Read profile.yaml → pipeline + domain hints.

2. GENERATE   Dispatch each pipeline stage. Within a stage, parallel
              sub-agents run in a single message. Each subagent has
              fresh context and reads its format file from the profile.

3. REVIEW     Dispatch reviewer (fresh context) on all artifacts + tree.

4. APPLY      Artifact issues → re-run affected gen-* subagents. Max 2
              GENERATE→REVIEW cycles.

5. RETURN     Tree gaps → return as proposed `?` items to caller
              (gather picks them up via on_ready return).
```

## Algorithm

### 1. LOAD profile

Read `skills/project-architecture/profiles/{{PROFILE}}/profile.yaml`.

It defines:
- `pipeline` — stages with the `gen-*` subagents to dispatch and output paths.
- `domain` — language, test framework, file extensions. Passed into subagent prompts.
- `antipatterns_url` — default for risks generation.

### 2. GENERATE

For each stage in `pipeline`:
- **Sequential stages** (`stage_1_sequential`, etc.): dispatch one at a time, wait for each.
- **Parallel stages** (`stage_2_parallel`): dispatch ALL in a single message with multiple Agent tool calls.
- **Final stage**: same as sequential, runs last.

For each `gen: X` entry:
- Read `skills/project-architecture/gen-X.md` as the subagent prompt.
- Substitute placeholders:
  - `{{DATA_FILE}}`, `{{REQUIREMENTS_FILE}}`, `{{PROFILE_DIR}}` (= `skills/project-architecture/profiles/{{PROFILE}}/`), `{{ARTIFACTS_DIR}}`, `{{DETAILS_DIR}}`
  - `{{DOMAIN}}` (injected as a short paragraph from `profile.domain`: "Language: python, tests: pytest, ...")
  - `{{OUTPUT}}` — the **current** entry's output. Derived from `writes:`, prefixed with `{{ARTIFACTS_DIR}}/`. For single-file outputs this is one path; for glob patterns (e.g. `interfaces/*.py`) this is the pattern inside the artifacts dir; for multi-output entries this is a list.
  - **Named prior-output placeholders** — for every pipeline entry `gen: Y` that ran before this one, expose a placeholder `{{<Y_UPPER>_OUTPUT}}` carrying Y's `writes:` path(s). Replace `-` with `_`, uppercase. Examples:
    - `gen: components` → `{{COMPONENTS_OUTPUT}}`
    - `gen: call-diagrams` → `{{CALL_DIAGRAMS_OUTPUT}}`
    - `gen: interfaces` → `{{INTERFACES_OUTPUT}}` (glob pattern of interface files)
    - `gen: state-machines` → `{{STATE_MACHINES_OUTPUT}}` (optional; empty string if skipped)
  Later generators reference prior outputs ONLY through these named placeholders when specifying reads or writes.
- Dispatch as a subagent with **opus** model, fresh context.

**`writes:` is the single source of truth for every artifact path.** Renaming an artifact in a profile propagates automatically through `{{<NAME>_OUTPUT}}` placeholders. Rules for gen-*.md prompts:

- **Read/write path references** (anywhere a subagent is told to read or write a file) MUST use placeholders — `{{OUTPUT}}` for its own output, `{{<PRIOR>_OUTPUT}}` for prior artifacts. No hard-coded paths, no `{{ARTIFACTS_DIR}}/components.md` shortcuts.
- **Prose references to artifacts by concept or generic filename** (e.g. "the components artifact", "call-diagrams.md" inside task descriptions, examples in check rules) are allowed. They describe the artifact's role, not a path to read. Profiles are expected to follow generic filename conventions for these to remain readable; a profile that aliases filenames still works because the actual read/write paths go through placeholders.

After each stage, the orchestrator verifies that the expected `writes:` paths exist on disk; missing files are a hard error. Verification semantics by `writes:` shape:

- **Single file path** (e.g. `components.md`) — the exact file must exist.
- **Glob pattern** (e.g. `interfaces/*.py`, `specs/*.t.sol`) — the directory portion must exist AND at least one of: (a) a file matching the glob pattern, OR (b) a sentinel file `<glob-dir>/.none`. The sentinel signals "zero items — intentional, not an oversight".
- **Glob with zero valid items** — generator MUST write the sentinel `<glob-dir>/.none` containing a one-line explanation ("no interfaces — system has no components exposing a public surface"). A generator that cannot produce even a sentinel returns `fatal: <reason>`.
- **List** (`[a.md, b.md]`) — each entry verified by its own shape (single file or glob).

**Generators MUST always write their artifact.** `[GAP]` / `[CHOICE]` are **inline markers** that live inside the generated file (e.g. "`[GAP]: caller not decided`" in a cell of `access-control.md`), so `gen-gaps` can pick them up from disk. A generator that cannot fill in any content still writes the file with a one-line note plus `[GAP]` markers explaining what is missing.

A subagent's top-level `fatal: <reason>` return is reserved for **fatal inability to run** — e.g. a required upstream artifact missing, profile misconfiguration, unreadable tree. In that case the orchestrator stops the pipeline. It is NOT used for normal "architecture decision missing" gaps; those go inline as `[GAP]` / `[CHOICE]` markers in the file.

Accepted return shapes from each generator:
- `written: {{OUTPUT}}` — artifact written; inline `[GAP]` / `[CHOICE]` markers (if any) are inside the file.
- `written: {{OUTPUT}} (none)` — for `optional: true` entries that have nothing to produce; the file is STILL written and contains a one-line explanation.
- `fatal: <reason>` — pipeline cannot proceed; orchestrator halts.

`optional: true` only changes what the generator is expected to produce when there is nothing to decompose (a single-line file with "(none)" explanation). The **missing-file check still applies** — the file at `{{OUTPUT}}` must exist after the stage. If a profile wants to allow a fully-absent artifact, it must declare `optional_output: absent` on the entry explicitly; in that case the missing-file check is skipped.

Orchestrator does not re-read generated files — they are on disk for the next stage.

### 3. REVIEW

Dispatch `reviewer.md` subagent (fresh context). Reads: tree + all artifacts + specs + `{{PROFILE_DIR}}/formats/*.md` (for formatting rules). Returns:

```
ARTIFACT_ISSUES: N
  A1: [check] — <artifact>: <description>
  ...
TREE_GAPS: N
  G1: [check] — <description> — parent: AD-NNN — suggested question: ? <text>
  ...
TRACEABILITY: X% average
CLEAN: yes/no
```

### 4. APPLY

- **Artifact issues** → cascade re-run:
  1. From reviewer output, find the **earliest flagged entry** in the pipeline (use pipeline stage + position as ordering).
  2. Re-dispatch that entry AND every subsequent entry in the pipeline, preserving the profile's stage order and parallelism rules. (Entries before the earliest flag are untouched.)
  3. After the cascade completes, re-dispatch `reviewer.md`.
- Hard cap: **2 full GENERATE→REVIEW cycles**. After the cap, report remaining artifact issues as-is without another re-run.
- **Tree gaps** → do NOT modify the tree. Collect into a return payload.

Rationale: downstream artifacts are built from upstream outputs, so re-running only the flagged entry leaves derived artifacts stale. The "earliest flagged entry → end of pipeline" rule is conservative but simple and deterministic — no need to parse prompts for a dependency graph. If finer-grained re-runs are needed later, add an explicit `reads: [<entry>, ...]` field to pipeline entries and replace the conservative rule with graph-based invalidation.

### 5. RETURN

Return to caller the reviewer's `TREE_GAPS` list as proposed `?` items. The reviewer is the single aggregator: it merges its own 8-check findings with every row from `{{GAPS_OUTPUT}}` (deduplicated), so the orchestrator does NOT separately merge gaps.md here — just surface what the reviewer produced.

```
Proposed tree gaps (N items):

1. → [?] AD-NNN-parent: [question text]
   Source: reviewer check #X (or gaps.md/<origin-artifact>)
   Why: [reviewer's description]

2. → [?] ...
```

Plus summary: `artifacts written to <dir>, reviewer: <N> issues / <M> gaps, traceability <X>%`.

When called by gather via `on_ready`, gather picks up this list and runs the batch protocol. When called standalone, print the list and exit.

## Rules

- **Orchestrator dispatches, does not generate.** No file writes by orchestrator itself.
- **Fresh context per subagent.** Each Agent tool call starts clean.
- **Parallelize where profile allows.** Stages marked `_parallel` run as a single message with multiple Agent calls.
- **Always regenerate from scratch.** Do not preserve partial outputs between cycles.
- **Respect `[GAP]` / `[CHOICE]`.** Do not ask subagents to invent missing info.
- **Max 2 SUMMARIZE→REVIEW cycles.** After that, return with remaining issues noted.

## Standalone invocation

```
/project-architecture --profile=python-library <tree_path> [requirements_path]
```
